# Decisions

このドキュメントは、実装上の主要な設計判断と意図を記録します。

## なぜ全探索（1575通り）にしたか

- 要件は「Diff 最小の最適解保証」であり、近似解ではなく確定最適が必要
- 最大探索数は `10C8 * (8C4/2) = 1575` と小規模
- 実運用で十分高速（ミリ秒オーダー）で、アルゴリズムが読みやすい
- 将来の保守でも正しさを説明しやすい

## なぜ SQLite を採用したか

- 単一バイナリ運用と相性がよく、導入コストが低い
- RoomState・履歴・統計を再起動後も維持できる
- Docker 運用でもボリュームマウントで扱いやすい
- MVP で必要な整合性と検索性能を満たす

## 観戦ローテの設計意図

- 最優先は戦力バランス（Diff 最小）
- 同点/僅差の候補でのみローテーションを効かせ、公平性を向上
- `spectator_count` と `last_spectated_at` で連続観戦を緩和
- 「公平性」と「試合品質」のトレードオフを明示的に制御

## レート補正の簡易モデル採用理由

- 自己申告XPowerのブレを運用中に補正したい
- 複雑なレーティング方式（Elo/Glicko）より、説明可能性と実装コストを優先
- ルールを固定:
  - 勝ち `+10`
  - 負け `-10`
  - `[-200, 200]` にクランプ
- `effective_power = declared_xpower + rating_delta` により、過去成績を穏やかに反映

## ロック・クールダウン・defer 対応の意図

- Discord での連打や二重実行による状態破損を防ぐ
- room 単位ロックで同時更新を抑止
- `/make` と `/next` にクールダウンを設けて連投を抑制
- 長時間処理は defer 応答で Interaction の制約（応答期限）を回避
- 結果として、運用時の事故率を下げる
